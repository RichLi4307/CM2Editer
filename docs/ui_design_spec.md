# CM2Editer UI 设计规范与优化规划

> 版本：v1.0（2026-07-21）
> 性质：**设计交付文档**。本文档只规定"长什么样、多大、什么颜色、什么状态"，不规定具体 egui 写法。
> 实施者按本文档的数值与规则自行落地代码；本文档中所有尺寸单位为 **逻辑像素（px，egui 点）**，颜色为 **sRGB Hex**。
> 置信关系：本文档是 UI 视觉与交互的**目标规范**；与现有代码冲突时，以本文档为改造目标；与 AGENTS.md 的开发约束冲突时，以 AGENTS.md 为准。

---

## 1. 背景与设计目标

### 1.1 用户画像

- 游戏 Mod 作者，**非专业程序员**，不熟悉 IDE 类复杂界面。
- 长时间在画布上拖拽节点、填写参数，视觉疲劳是真实成本。
- 中文为主，英文/日文次之；中文字形比拉丁字母"显大"，需要更宽松的行高与间距。

### 1.2 现状审计结论

当前 UI（v0.3.0）为功能优先的深色界面，存在以下系统性问题：

| 问题类别 | 现状 | 影响 |
|----------|------|------|
| 常量分散 | 尺寸/颜色散落在 `app.rs`、`node_renderer.rs`、`edge_renderer.rs`、各面板文件内 | 无法统一调整，改一处漏十处 |
| 层级缺失 | 背景仅 `#1E1E1E` + 节点 `#2D2D2D` 两层，面板/卡片/弹窗无视觉分层 | 用户分不清"什么浮在什么上面" |
| 控件规格不一 | ComboBox 宽度有 100/120/160/180/clamp(80,160) 五种取值；卡片尺寸 64×40 / 130×40 / 130×46 / 140×50 四种 | 界面显得"拼凑" |
| 反馈薄弱 | 无 Toast；状态消息只在底栏一闪而过；成功/失败无色彩区分 | 用户操作后不确定结果 |
| 弹窗不统一 | 三种选择器尺寸各异、无统一按钮区模板 | 学习成本高 |
| 文本无层级 | 字号只有 12/11/10 三档，面板标题与正文几乎无区分 | 信息密度高但可读性差 |

### 1.3 设计目标

1. **一致性**：所有控件从同一套设计令牌（Design Tokens）取值，禁止散落硬编码。
2. **分层感**：通过 4 级背景色 + 边框/阴影规则，让用户一眼识别"底层画布 / 固定面板 / 浮层弹窗"。
3. **可读性**：建立 6 级字号 + 3 级文本色层级，中文优先的宽松行高。
4. **反馈闭环**：每个用户操作都有可见反馈（hover、按下、成功、失败、进行中）。
5. **egui 可行性**：所有规则在 egui 即时模式下可落地，不依赖 egui 不支持的效果（如真实投影用"深色描边 + 半透明底色"模拟）。

## 2. 设计令牌（Design Tokens）

> 实施要求：全部令牌集中到 `src/ui/theme.rs`（可拆分为 `theme/tokens.rs`），业务代码只允许引用令牌，禁止再出现魔法数字。现有 `Theme::BACKGROUND` 等常量映射到令牌并保留别名过渡。

### 2.1 间距体系（4px 基准网格）

| 令牌 | 值 | 用途 |
|------|-----|------|
| `space_1` | 4 | 图标与文字间距、紧凑行内间距 |
| `space_2` | 8 | 控件之间的默认垂直间距、面板内边距（小） |
| `space_3` | 12 | 面板内边距（标准）、分组间距 |
| `space_4` | 16 | 面板区块间距、弹窗内边距 |
| `space_5` | 24 | 弹窗区块间距、大区块分隔 |
| `space_6` | 32 | 页面级留白 |

规则：任何两个元素的间距必须命中上表，禁止 5/6/7/10/14 等中间值。

### 2.2 圆角体系

| 令牌 | 值 | 用途 |
|------|-----|------|
| `radius_small` | 4 | 按钮、输入框、下拉框、复选框、小卡片 |
| `radius_medium` | 6 | 选择器弹窗内的卡片（坐标卡、命名空间卡、state 卡） |
| `radius_large` | 8 | 节点卡片、弹窗窗口、欢迎卡片 |
| `radius_round` | 完全圆形 | 端口圆点、色点、徽标 |

> **egui 0.31 落地约束**：`CornerRadius` 各角为 `u8`，**最大 255**，不存在 `999` 这种"足够大即圆"的写法。完全圆形的实现只有两条合法路径：(a) 用 `painter.circle_filled / circle_stroke` 直接画圆；(b) 圆角取 `min(width, height) / 2` 且 ≤ 255。实施时禁止写魔法大数。
> 现状：节点圆角 8 ✔ 保留；选择器卡片 4 → 升为 6；欢迎卡片 12 → 降为 8（与节点统一）。

### 2.3 字号与文本层级

| 令牌 | 字号 | 字重 | 颜色令牌 | 用途 |
|------|------|------|----------|------|
| `text_display` | 22 | Bold | `text_primary` | 欢迎卡片标题（现状 22 ✔） |
| `text_heading` | 16 | Bold | `text_primary` | 属性面板节点标题、弹窗标题 |
| `text_title` | 14 | Bold | `text_primary` | 面板分区标题、工具栏分组名 |
| `text_body` | 13 | Regular | `text_primary` | 正文、按钮文字、输入框文字 |
| `text_caption` | 12 | Regular | `text_secondary` | 参数说明、次要信息、状态栏 |
| `text_micro` | 11 | Regular | `text_secondary` | 节点端口标签、参数预览、类型徽标 |

行高：正文 = 字号 × 1.5（中文宽松），列表行高见第 4 节。

> 现状对照：节点标题 12 → 13（`text_body`）；端口标签/参数预览 11 ✔（`text_micro`）；画布信息覆盖层 14 → `text_title`。

### 2.4 颜色体系

#### 2.4.1 背景层级（Elevation）

| 令牌 | Hex | 用途 |
|------|-----|------|
| `bg_app` | `#1E1E1E` | 画布背景（现状 ✔ 保留） |
| `bg_panel` | `#252526` | 侧栏、底部面板、工具栏、状态栏 |
| `bg_card` | `#2D2D2D` | 节点卡片、列表选中卡片（现状 ✔） |
| `bg_elevated` | `#383838` | 弹窗、右键菜单、下拉列表、Toast |
| `bg_overlay` | `#1E1E1E` @ 60% 透明 | 模态弹窗背后的遮罩（egui 用半透明填充模拟） |

规则：层级越高越亮，任何浮层必须比其下层面板亮至少一档。

#### 2.4.2 边框与分隔

| 令牌 | Hex | 用途 |
|------|-----|------|
| `border_subtle` | `#3C3C3C` | 面板内分隔线、输入框默认边框 |
| `border_default` | `#505050` | 节点边框（现状 ✔）、卡片边框 |
| `border_strong` | `#6B6B6B` | 输入框 hover 边框 |
| `border_focus` | `#2196F3` | 输入框 focus、选中描边 |

#### 2.4.3 文本色

| 令牌 | Hex | 对比度（对 `bg_panel`） | 用途 |
|------|-----|------|------|
| `text_primary` | `#FFFFFF` | 16.9 | 正文（现状 ✔） |
| `text_secondary` | `#B4B4B4` | 8.2 | 辅助文本（现状 #B4B4B4 ✔） |
| `text_disabled` | `#6B6B6B` | 3.4 | 禁用文本，仅用于非关键信息 |
| `text_link` | `#64B5F6` | — | 可点击链接（状态栏错误计数等） |

#### 2.4.4 语义色

| 令牌 | Hex | 用途 |
|------|-----|------|
| `accent` | `#2196F3` | 主强调色：选中发光、focus 框、主按钮、分割条 hover（现状 ✔ 统一为同一蓝） |
| `success` | `#4CAF50` | 保存成功、验证通过、框选 Crossing 模式（现状 ✔） |
| `warning` | `#FF9800` | 警告徽标、端口已占用提示（现状 ✔） |
| `error` | `#F44336` | 错误边框、验证错误、非法连线（现状 ✔） |
| `entry_amber` | `#FFC107` | 入口钉（现状 ✔ 保留，作为品牌点缀色） |

#### 2.4.5 功能色（保持不变，仅登记）

- 节点分类色 11 种、`scene_category_color` 7 种、端口类型色：维持 `theme.rs:51-122` 现状，**不改色相**（用户已有肌肉记忆）。
- 连线：`wire_default #C8C8C8`、`wire_invalid = error`、`wire_occupied = warning`。
- 画布网格：`grid #323232`（现状 ✔）。

#### 2.4.6 状态透明度规则

| 状态 | 实现 |
|------|------|
| 禁用/不兼容（灰显） | 元素整体 alpha × 0.25（现状连线灰显规则 ✔，推广到按钮/菜单项） |
| Hover（深色底控件） | 背景色向 `#FFFFFF` 混合 8% |
| Pressed | 背景色向 `#FFFFFF` 混合 16% |
| 选中发光 | `accent` 色描边外扩 4px，alpha 60%（现状 ✔） |

## 4. 控件规范

### 4.1 按钮

三种语义按钮 + 一种工具栏变体：

| 类型 | 背景 | 文字 | 用途 |
|------|------|------|------|
| 主按钮（Primary） | `accent #2196F3`，hover 白 8% 混合 | `#FFFFFF` | 弹窗确认、导出等唯一主动作，每屏最多 1 个 |
| 次按钮（Secondary） | `bg_card #2D2D2D` + 1px `border_default` | `text_primary` | 普通操作（取消、新建等） |
| 危险按钮（Danger） | `bg_card` + 1px `error` 描边 | `error #F44336` | 删除、清空等不可逆操作 |
| 工具栏按钮（Ghost） | 透明，hover 白 8% 混合 | `text_primary` | 顶部工具栏、图标按钮（收藏星标等） |

统一尺寸：**高 28**，圆角 `radius_small` (4)，水平内边距 12，图标与文字间距 `space_1` (4)。禁用态：整体 alpha 40%，不响应 hover。

### 4.2 下拉框（ComboBox）

**形状规范**（egui ComboBox 需通过 `ComboBox::width` + 自定义 `ui` 或全局 style 达成）：

| 项 | 值 |
|----|-----|
| 高度 | 28 |
| 圆角 | `radius_small` (4) |
| 边框 | 1px `border_default`；hover `border_strong`；展开时 `border_focus` |
| 右侧箭头 | 下三角 ▾，展开时旋转 180° 为上三角 ▴，颜色 `text_secondary` |
| 弹出列表 | 背景 `bg_elevated #383838`，圆角 4，1px `border_default` 描边；列表项高 24，左右内边距 8；hover 项白 8% 混合；选中项左侧 2px `accent` 竖条 |
| 最大弹出高度 | 300，超出滚动（现状搜索窗口 300 ✔ 推广） |

> **egui 0.31 落地约束（防呆必读）**：
> 原生 `ComboBox::from_label` 的弹出列表**无内置最大高度**，长列表会直接冲出屏幕。规范实现必须用 `ComboBox::show_ui` + 内部 `ScrollArea::vertical().max_height(300.0)`，禁止裸 `selected_text` 快速用法处理长列表。
> 原生箭头**不可旋转、不可改色**（跟随 `Visuals`）。旋转箭头属"需自定义绘制"项：用 `Painter` 自绘箭头时，整个下拉框应抽成公共组件 `TokenComboBox`，禁止每个调用点各自画一份。
> 统一封装：阶段 B 必须先产出 `TokenComboBox` 公共组件（宽度档位、箭头、滚动、选中竖条全部内置），所有调用点只允许传 `DropdownWidth` 枚举，禁止传裸数值。

**宽度规则——只允许三档**（消除现状 5 种取值）：

| 档位 | 宽度 | 适用场景 |
|------|------|----------|
| 紧凑 `dropdown_sm` | 100 | 数据源类型切换等短选项（现状 100 ✔ 归入此档） |
| 标准 `dropdown_md` | 160 | 枚举参数、条件模板（现状 120/160 → 统一 160） |
| 宽松 `dropdown_lg` | 200 | 方法名、NPC 列表等长文本（现状 180 → 微调到 200） |

自适应场景（节点库分类过滤）：`available_width.clamp(120, 200)`（现状 clamp(80,160) 太窄，长分类名截断）。语言选择器固定 96 属例外。

### 4.3 文本输入框

| 项 | 值 |
|----|-----|
| 高度 | 28（单行）；多行最小高 72 |
| 圆角 | 4 |
| 边框 | 1px `border_default`；hover `border_strong`；focus `border_focus` 且外扩 1px 发光 |
| 内边距 | 左右 8 |
| 占位符 | `text_disabled #6B6B6B` 斜体 |
| 非法输入 | 边框 `error`，下方 4px 间距后接 12px `error` 色错误文本 |
| 宽度 | 属性面板内：默认占满可用宽；与下拉框并排时 `min(120, 剩余宽)` |

强制规则（沿用 AGENTS.md）：参数编辑一律用 `ParamTextEdit`，禁止裸 `text_edit_singleline`。节点库搜索框属过滤器，可例外，但样式必须与参数输入框一致。

### 4.4 复选框 / DragValue

- 复选框：18×18，圆角 4，未选中 1px `border_default` 描边 + `bg_card` 底；选中 `accent` 底 + 白色 ✓。
- DragValue：高 28，圆角 4，`bg_card` 底 + 1px `border_default`；拖拽光标 `ResizeHorizontal`；hover 时显示 `‹ ›` 提示箭头。Vector/xyz 三字段并排，字段间距 `space_1` (4)，单字段宽 ≥ 56。

### 4.5 分隔线与分区标题

- 面板内水平分隔线：1px `border_subtle`，上下 `space_3` (12) 间距。
- 分区标题（如"常用参数""高级参数"）：`text_title` 14 Bold，下方 4px 间距；可折叠分区右侧箭头 ▸/▾。

### 4.6 滚动区

- 滚动条宽 8，滑块 `border_default`，hover `border_strong`，轨道透明。
- 列表项选中：整行 `accent` @ 16% 透明底 + 左侧 2px `accent` 竖条。

## 8. 反馈与状态

### 8.1 Toast 通知系统（当前缺失，P1 新增）

| 项 | 值 |

## 12. egui 架构防呆（coder 必读，优先级高于前文任何视觉细节）

### 12.1 能力可行性对照表

| 规范项 | egui 0.31 可行性 | 落地方式 |
|--------|------------------|----------|
| 面板固定高度/宽度范围 | ✅ 原生 | `TopBottomPanel::exact_height` / `SidePanel::width_range` |
| 模态遮罩 | ✅ 原生 | `egui::Modal`（0.31 已内置），禁止手绘遮罩 |
| Toast 浮层 | ⚠ 需自定义 | `egui::Area` + `Order::Foreground` + 锚定 CentralPanel rect；不引入第三方 crate（AGENTS.md 约束） |
| ComboBox 滚动弹出列表 | ⚠ 需自定义 | `ComboBox::show_ui` + `ScrollArea::max_height`（见 4.2 节注） |
| 箭头旋转/改色、选中竖条 | ⚠ 需自定义绘制 | 封装进 `TokenComboBox`，禁止散落 |
| 输入框 focus 发光 | ⚠ 半原生 | `Visuals` 只能改 stroke 色；"外扩发光"需按 `response.rect` 自绘，允许降级为仅描边变色（登记偏差即可） |
| 占位符斜体 | ❌ 不支持 | `hint_text` 样式不可控，**放弃斜体**，仅颜色区分（属已批准偏差） |
| 圆角 999 | ❌ 不存在 | `CornerRadius` 上限 255，圆形一律 `circle_filled` 或 `min(w,h)/2`（见 2.2 节注） |
| 标题栏渐变压暗 | ⚠ 可选 | `Painter::rect_filled` 两次叠加；属可选优化，可跳过 |
| 分段控件、列头、徽标 | ⚠ 需自定义 | 常规 `ui.horizontal` + `Painter` 即可，无坑 |

### 12.2 强制守卫（用测试把规范变成编译期/CI 约束）

项目已有先例（`test_boolean_output_nodes_are_in_conditions` 强制节点分类），UI 规范采用同样的"测试即护栏"策略：

1. **颜色守卫测试**：扫描 `src/ui/**`（`theme` 模块除外），出现 `Color32::from_rgb` / `from_rgba` 裸调用即测试失败。放行名单：测试模块、i18n 默认色回退。
2. **尺寸守卫测试**：`DropdownWidth` / 卡片尺寸 / 行高等令牌以 `const` 定义后，测试断言其值属于规范集合（如下拉宽 ∈ {96, 100, 160, 200}）。防止有人"顺手"加个 175。
3. **间距守卫**：不强制扫描（收益低），靠 code review + 令牌命名约定（变量必须叫 `space_*`，裸 `ui.add_space(7.0)` 在 review 中一眼可辨）。
4. **i18n 守卫测试**：遍历代码中所有 `i18n.t("...")` 的 key（或维护 key 清单），断言 zh/en/ja 三个 JSON 均存在该 key。缺失即失败——防止新 UI 文案只加了中文。

### 12.3 Widget ID 规则（本项目已踩过 `duplicate widget ID` 坑）

- 循环内创建的任何 widget **必须** `id_salt` 带稳定业务 key（节点 id + 参数 key），**禁止**用循环下标（删除元素后 ID 漂移）。
- 同一参数在多个面板/弹窗同时可见时（如属性面板 + 条件编辑器），`id_salt` 须加面板前缀区分。
- 弹窗型编辑器内的 widget ID 生命周期 = 弹窗打开期间，关闭后状态（文本缓冲等）必须清理，防止下次打开出现幽灵焦点。

### 12.4 样式作用域规则

- 禁止用 `ctx.set_style` 做局部样式（全局泄漏，下一帧影响所有面板）。局部样式只允许：(a) `egui::Frame` 包裹；(b) `ui.scope_builder` / 克隆 `Style` 后局部替换；(c) `Painter` 直接绘制。
- `Visuals` 全局调整只允许在 `app.rs` 启动时装配一次（深色基调 + 令牌映射），运行期禁止再改。
- 自定义组件（`TokenComboBox` / `TokenDialog` / Toast）一律通过参数或令牌取色，禁止读 `ui.visuals()` 里的硬编码色。

### 12.5 令牌迁移规则（阶段 A 防半成品）

- **禁止 `#[deprecated]` 别名过渡**：弃用警告会打破 `cargo clippy 0 warnings` 提交门槛，导致别名永远清不掉。
- 正确做法：阶段 A 在**同一个 commit** 内完成"新令牌定义 + 全部调用点替换 + 旧常量删除"，让旧名字直接编译错误（Rust 编译器就是最好的防呆）。
- 阶段 A 验收追加：`grep -rn "from_rgb\|from_rgba" src/ui --exclude-dir=theme` 无业务代码命中。

### 12.6 i18n 强制规则

- 本规范新增的所有文案（Toast 四类、空状态、列头、按钮、占位符）必须先定义 key 再写代码，key 命名沿用现有 `label.*` / `panel.*` 前缀风格。
- zh/en/ja 三份 JSON 必须同步补齐；允许 ja 暂用英文占位（现状已有此惯例），但不允许缺 key（缺 key 会显示原始 key 字符串，属 P3.1 已修过的缺陷类别）。

### 12.7 坐标与缩放防呆

- 本规范所有 px 为 **egui 逻辑点**。禁止任何代码乘/除 `pixels_per_point` 或 `zoom_factor`；HiDPI 由 egui 自动处理。
- Toast、缩放徽标等浮层位置必须从对应 Panel 的 `rect` 派生，禁止用 `ctx.screen_rect()`（会盖住侧栏/工具栏）。

### 12.8 性能防呆（画布热路径）

- `update_canvas` 每帧执行：禁止在节点/连线渲染循环内做堆分配（`String::new`、`format!`、`Vec::collect`）、i18n 查找之外的 map 查询可以、但禁止正则/解析。
- 现有可见区裁剪（`cull_margin = 50`）必须保留；新增装饰元素（端口描边、⚠ 徽标、箭头）不得绕过裁剪逻辑。
- 文本绘制优先复用 `Painter::text` 的 `Galley` 缓存路径，禁止每帧 `layout` 同一字符串多次。

### 12.9 焦点与文本输入（沿用既有约束）

- 参数文本框一律 `ParamTextEdit`（AGENTS.md 强制，本项目已踩过 ComboBox 后失焦的坑）。
- 新文本输入场景（Toast 无；搜索框已有）若出现在 ComboBox 之后的布局顺序，必须先验证焦点保持，再合入。

### 12.10 中文排版下限

- 中文正文最小 12px（`text_caption`）；`text_micro` (11px) 仅限拉丁字符为主的场景（NodeType 名、坐标数字）。端口标签若显示中文参数名，需用 12px 并在规范中按 `text_caption` 处理（实施时如发现端口标签含中文，自动升级字号并登记）。

## 15. 吸附式窗口面板方案（待架构师审核）

> 状态：候选架构方案，尚未实现。本章用于记录把左栏、右栏、底栏从 egui 布局面板改为「吸附式窗口」的构想，供架构师与设计师一并评估。

### 15.1 背景与动机

当前主界面采用 egui 标准布局：

```text
┌───────────────── 顶部工具栏 ─────────────────┐
├──────┬──────────────────────────┬────────────┤
│ 左栏 │        中央画布          │   右栏     │
├──────┴──────────────────────────┴────────────┤
│              底部三列面板                     │
├──────────────────────────────────────────────┤
│                  状态栏                       │
└──────────────────────────────────────────────┘
```

该布局存在以下结构性问题：

1. **底栏与侧栏耦合**：`TopBottomPanel::bottom("bottom_main")` 在 `SidePanel::left` / `SidePanel::right` 之后创建，因此底栏被嵌在左右侧栏之间的中央区域。左栏变宽时，底栏宽度变窄，内部两条分割线会跟随漂移。
2. **侧栏信息密度被高度挤压**：左栏需要承载节点库、工程树、命名空间、坐标、概览等多个标签页，信息量大；而当前左栏高度被中央画布 + 底栏挤压，无法充分利用垂直空间。
3. **扩展性差**：后续若增加多工程树、节点库常驻、更多底栏工具页等，固定布局难以灵活扩展。

### 15.2 方案目标

- 将左栏、右栏、底栏从「布局系统的一部分」改为「独立的 egui 窗口实例」。
- 这些窗口默认吸附到屏幕边缘，但保留窗口身份，可独立设置尺寸、位置、吸附状态。
- 各栏尺寸互不耦合：左栏可以足够宽，底栏宽度不再受左栏影响。
- 中央画布使用完整窗口，面板浮于画布之上，点击事件由面板优先消费。

### 15.3 整体布局

```text
窗口 Z 序（从底到顶）：

1. 中央画布（CentralPanel，全屏）
2. 顶部工具栏（TopBottomPanel::top，全宽）
3. 状态栏（TopBottomPanel::bottom，全宽）
4. 底栏三列窗口（egui::Window，底部吸附，全宽）
5. 左栏窗口（egui::Window，左侧吸附）
6. 右栏窗口（egui::Window，右侧吸附）
7. 弹窗 / 模态（egui::Window / egui::Modal）
```

> 说明：状态栏和顶部工具栏仍保持为 `TopBottomPanel`，因为它们是强全局控件，不适合浮动。左/右/底栏改为窗口。

### 15.4 默认几何与吸附规则

| 窗口 | 默认吸附位置 | 默认尺寸 | 最小尺寸 | 最大尺寸 | 可浮动 |
|------|--------------|----------|----------|----------|--------|
| 左栏 | 左侧，顶部对齐工具栏下沿 | 宽 240，高 占满可用垂直空间 | 宽 200，高 200 | 宽 400，高 无 | 是 |
| 右栏 | 右侧，顶部对齐工具栏下沿 | 宽 280，高 占满可用垂直空间 | 宽 240，高 200 | 宽 500，高 无 | 是 |
| 底栏 | 底部，左对齐窗口左沿 | 高 240，宽 占满窗口宽 | 高 140，宽 400 | 高 480，宽 无 | 是 |

**吸附判定（snap）**：

- 当窗口标题栏被拖拽，且窗口中心距某条屏幕边缘的距离小于阈值（建议 40 px）时，释放后自动吸附到该边缘。
- 吸附时，窗口恢复到该边缘的默认尺寸；如果窗口本身已被用户手动调整过，则优先保持用户尺寸，仅对齐位置。
- 可提供一个「锁定吸附」开关，避免误拖；默认锁定。

### 15.5 与画布的层级关系

- 画布使用 `CentralPanel::default()` 铺满整个窗口，**不再为面板留空**。
- 面板窗口在 `Order::Middle` 或更高层级绘制，天然遮挡画布。
- 鼠标事件由 egui 的窗口系统处理：点击面板窗口不会穿透到画布。
- 当面板窗口尺寸变化时，画布无需重新计算可见区域；画布始终全屏渲染，但用户只能看到未被遮挡的部分。

### 15.6 持久化

- 每个窗口保存：是否吸附、吸附位置、窗口尺寸、浮动时的位置。
- 保存路径：`%APPDATA%/CM2Editer/settings.json` 的 `docked_windows` 字段。
- 启动时读取；若未找到或数值非法，回退到默认吸附布局。

### 15.7 对现有代码的影响

| 文件 | 当前实现 | 改动方式 |
|------|----------|----------|
| `src/app.rs` | 使用 `SidePanel::left/right` 和 `TopBottomPanel::bottom` | 改为创建 `egui::Window` 并管理窗口状态；移除与侧栏宽度耦合的底栏约束 |
| `src/ui/panels/mod.rs` | 各面板函数直接接收 `&mut Ui` | 保持入口不变，但调用点由 `SidePanel::show` 改为 `Window::show` |
| 新增 `src/ui/docked_window.rs` | 无 | 封装吸附窗口：默认位置、吸附判定、拖拽检测、状态持久化 |
| `src/settings.rs` | 保存语言、收藏、最近节点等 | 新增 `DockedWindowSettings` 数据结构 |

### 15.8 需要解决的问题

1. **Z 序与焦点**：
   - 点击左栏时，左栏是否应提升到最顶层？还是固定层级（左/右/底）？
   - 建议：固定层级，避免窗口互相遮挡导致混乱；仅在拖动时临时提升。

2. **底栏与左右栏重叠**：
   - 左栏底部会与底栏左侧重叠，这是窗口化布局的正常现象。
   - 重叠区域的内容可通过滚动或调整窗口尺寸解决；不应强制压缩任何一栏。

3. **全屏/最大化适配**：
   - 窗口最大化时，各窗口应重新计算吸附位置。
   - 浮动窗口不应被移出屏幕外。

4. **触控 / 小屏幕**：
   - 在较低分辨率下，三栏可能无法同时容纳。可提供「隐藏侧栏」或「自动堆叠」策略。

5. **与拖拽节点的冲突**：
   - 从节点库拖节点到画布时，若鼠标经过面板窗口，窗口会接收拖拽事件。需要确保画布侧的拖拽落点逻辑仍能从全局鼠标位置正确计算，而不是被面板遮挡。

### 15.9 与实施路线图的关系

- 若架构师审核通过，本方案将替代第 10 章「实施路线图」中阶段 B/C/D 的部分布局相关任务。
- 阶段 A（令牌化）已完成的 `tokens` 体系仍然适用；窗口的边框、标题栏、按钮等控件同样从 `tokens` 取值。
- 原第 9 章「暂不做」中的「面板自由停靠/多窗口」条目将取消，并拆分为新的 P1/P2 任务：
  1. 实现吸附窗口容器（P1）。
  2. 迁移左栏、右栏、底栏为窗口（P1）。
  3. 持久化与吸附恢复（P2）。
  4. 小屏幕与触控适配（P2）。

### 15.10 工作量与风险

- **工作量**：中高。核心 docking 逻辑需手写，且 `app.rs` 布局代码需较大重构。
- **风险**：egui 0.31 的窗口系统虽然支持任意位置/尺寸，但没有内置 docking 吸附，手动实现需测试大量边界情况（窗口重叠、拖拽、吸附、恢复等）。
- **替代方案**：如果认为自研 docking 风险过高，可评估引入 `egui_dock` 等第三方 docking 库，但需评估其与当前 egui 0.31 版本的兼容性、许可证、API 侵入性。

---

> **结论**：本方案旨在通过「窗口化 + 边缘吸附」彻底解决左栏/底栏尺寸耦合问题，同时保留 egui 的即时渲染优势。是否采纳、是否分阶段实施、是否引入第三方库，均需架构师决策。

### 16.4 架构防呆缺口

当前防呆体系只有「颜色守卫」一根支柱，对 egui 项目不够。

1. **颜色守卫扫描范围不足**：`no_bare_color32_outside_theme` 仅扫描 `src/ui/**`，但 `src/app.rs` 在 `src/` 根下，不在扫描范围。建议扩展为 `src/ui` + `src/app.rs`（或把 UI 主布局代码移入 `src/ui`）。

2. **缺少尺寸/间距守卫测试**：`tokens` 已定义 `SPACE_*`、`RADIUS_*`、`TEXT_*`，但代码中仍散落 `12.0`、`24.0`、`4.0` 等。建议：

3. **缺少 i18n 守卫测试**：欢迎页 key 已三语齐全，但未来 Toast、空状态、列头还会新增 key。建议添加 `test_all_i18n_keys_present`，遍历代码中 `i18n.text` / `i18n.format` 使用的 key，检查 zh/en/ja 均存在。

4. **Widget ID 规则无自动化检查**：循环内 widget 的 `id_salt` 仍依赖人工审查。短期内可在 PR 清单中强制审查，长期可写静态脚本扫描。

### 16.5 对第 15 章吸附式窗口方案的 UX 评估

**优点**
- 能根治左栏/底栏尺寸耦合问题。
- 中央画布可获得完整窗口，渲染逻辑简化。
- 吸附状态持久化后，用户可自定义布局。

**风险**
1. **重叠区域遮挡内容**：左栏底部与底栏左侧重叠时，用户可能不知道被压住的画布内容还存在。
2. **拖拽冲突**：从节点库拖节点到画布，若鼠标经过其他面板窗口，事件可能被面板截获，落点计算复杂。
3. **标题栏侵占垂直空间**：三个面板窗口各带标题栏，对非程序员用户会显得像 IDE，而非专注于节点编辑。
4. **小屏幕灾难**：在 1366×768 等低分辨率下，三栏同时吸附后画布可能只剩中间一条。
5. **误拖风险**：用户可能不小心把面板拖成浮动，找不到重新吸附的入口。

**建议**
- **不要将本方案纳入当前 P3.4 路线图**，应放到 v0.4.0 分支试点，避免阻塞发布准备。
- 若一定要试点，必须补充：一键重置吸附布局、默认锁定吸附、小屏幕自动隐藏策略、拖拽节点时的面板穿透逻辑。
- 替代方案：先评估 `egui_dock` 等第三方 docking 库，比较自研与引入的工期与稳定性。
